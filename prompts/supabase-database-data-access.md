# Biasly Supabase Database and Data Access Implementation

## Goal

Establish the production-ready Supabase persistence foundation for Biasly News: an initial SQL schema for the required pipeline tables, typed TypeScript database definitions, a server-only Supabase service-role client, and small reusable query modules. This increment must not add scraping, scheduler, AI analysis, pgvector, API routes, or replace the existing fixture-based UI.

## Decisions and assumptions

- This is the initial database/data-access layer only. News pages remain untouched and continue rendering their current fixtures until a dedicated UI-data integration increment is approved.
- Use an imperative, dashboard-compatible canonical schema at `supabase/schema.sql`, as explicitly required by `AGENTS.md`. No remote database changes will be performed automatically because no linked Supabase CLI project or authenticated Supabase MCP server is configured.
- Add the stable `@supabase/supabase-js` package through npm and commit the updated lockfile. Keep the project’s existing TypeScript 5 toolchain, which satisfies the current Supabase SDK requirement.
- Use UUID primary keys generated by `gen_random_uuid()`, `timestamptz` audit fields, foreign keys, validity checks, unique dedupe constraints, and indexes that support the planned pipeline reads.
- Create the required tables: `sources`, `articles`, `article_analyses`, `logs`, `oxylabs_schedules`, and `oxylabs_schedule_runs`. Do not add the pgvector extension, embedding column, or related-article query in this increment; the project explicitly defers that work until after AI analysis.
- `sources` records publisher configuration (name, listing URL, parser strategy, active flag, optional logo). Do not seed publisher records or hardcode URLs.
- `articles` is append-only at the application level; enforce unique `original_url` and non-null image/published/raw-text fields. Include a unique canonical URL when present using a partial unique index.
- `article_analyses` has one analysis per article and includes the specified summary, sentiment, bias/framing, confidence, notes, loaded terms, disclaimer, and model fields. Add checks for score/percentage/confidence ranges and a percentage-sum constraint; compute `bias_score` as a stored generated column from left/right percentages so the formula cannot drift.
- `logs` records pipeline events with a level, event name, optional source/article/schedule references, message, structured metadata, and timestamp.
- Scheduler tables preserve Oxylabs large IDs as `text`, never JavaScript `number`, to avoid 64-bit precision loss.
- Enable RLS on all public tables. Do not create browser-access policies or grant public table access yet: initial access is server-side only through the service-role client, while the public UI continues using static fixtures. The SQL must explicitly revoke `anon` and `authenticated` privileges from these tables; future public read access requires a deliberate policy/GRANT review.
- The only Supabase client introduced here is a server-only service-role client. It reads `NEXT_PUBLIC_SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` at runtime, validates they are present without printing them, and must never be imported by browser/client modules.
- Add a `.env.example` using empty placeholders for the two Supabase variables and the existing Clerk variables required by the app. Do not copy, print, or alter the ignored local `.env` file. Keep the environment table in `AGENTS.md` unchanged because no variable names are being changed.
- Query modules will be small, typed, and server-only. They will expose foundational reads: active sources; URL existence checks in chunks no larger than 15; articles with their source and optional analysis for future UI use; an article by ID; an article by original/canonical URL; pending analysis using the required left-join semantics; and recent logs/scheduler records. They will not add writes, mutations, API route handlers, pipeline orchestration, or UI wiring.
- For joined reads, avoid filtering with `.eq('foreignTable.column', value)` as required by `AGENTS.md`.

## Implementation requirements

1. Install `@supabase/supabase-js` using npm with the lockfile updated. Do not add `@supabase/ssr`, Supabase Auth, or an anon/browser client.
2. Add `npm run typecheck` as `tsc --noEmit` to match the project’s documented required checks.
3. Add `supabase/schema.sql` with PostgreSQL SQL that can be safely executed once in the Supabase SQL Editor for a new Biasly project. It must enable the `pgcrypto` extension, create the six required tables, add all keys/checks/indexes/RLS/privilege revocations described above, and never include seed publisher URLs.
4. Write `lib/supabase/types.ts` as the database contract matching every schema column and the Supabase `Database` generic shape used by `createClient`. Use explicit row/insert/update types and avoid `any`.
5. Write `lib/supabase/server.ts` with `import "server-only"`, a `getSupabaseServerClient()` factory based on `createClient<Database>()`, and a configuration error that names missing environment variable keys only. Ensure the service-role key is never included in a public variable, response, log, or browser bundle.
6. Add query modules under `lib/supabase/queries/`, all importing the server client only. Return data or throw contextual errors that do not disclose credentials. Keep page view models separate from raw schema types when needed.
7. Cap every URL existence `.in()` query at 15 values and return a `Set<string>` of matching original/canonical URLs. Handle an empty input list without a database request.
8. Implement pending-analysis reads using a left join from `articles` to `article_analyses` and determine missing analyses from returned data rather than `articles.analyzed_at` alone.
9. Do not modify `app/page.tsx`, `app/news/[slug]/page.tsx`, Clerk configuration, `AGENTS.md`, existing fixture content, or unrelated configuration.
10. Do not execute the schema against a hosted database automatically. Provide the exact dashboard SQL Editor and local verification steps after implementation.

## Security requirements

- Never read, print, commit, or expose the contents of `.env`, `.env.local`, service-role keys, publishable keys, cookies, or tokens.
- The service-role client must be server-only and never imported into a Client Component, client action, or browser API.
- Enable RLS on every public-schema table and leave anon/authenticated without grants or policies in this foundation increment.
- Do not use Supabase Auth; Clerk remains the sole identity provider.
- Do not use `SECURITY DEFINER` functions, views, browser-side writes, or public mutation endpoints.
- Preserve the 64-bit precision of Oxylabs schedule/job IDs by storing and typing them as strings.

## Exact manual test steps expected after implementation

1. Create or open the intended Supabase project in the Supabase Dashboard.
2. Open **SQL Editor**, paste the complete contents of `supabase/schema.sql`, and run it once.
3. In **Table Editor**, confirm the six tables exist and that RLS is enabled for each one.
4. Confirm no publisher/source rows are pre-seeded; add sources only through a future approved setup or pipeline task.
5. Copy `.env.example` to an ignored local environment file and set `NEXT_PUBLIC_SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` from the project’s Connect/API settings. Do not place the service-role value in a `NEXT_PUBLIC_` variable.
6. Run `npm run typecheck`, `npm run lint`, and `npm run build` from the project root.
7. Start the app with `npm run dev`, then visit `http://localhost:3000` and `http://localhost:3000/news/peace-proposal`; both should continue to render the existing public fixture UI.
8. Before implementing scraping, run the future `/api/sources` route or a server-only query to list active sources and select source limits as required by `AGENTS.md`.
